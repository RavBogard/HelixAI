<objective>
## Goal
Build the full audit orchestrator that ties together all Phase 8-11 infrastructure: generate presets via mock harness, compare against reference corpus schemas, run intent/musical validation, and produce structured per-family compliance reports.

## Purpose
This is the culmination of the v5.0 audit infrastructure. Without this orchestrator, the mock harness, diff engine, validation, and schema extractor remain isolated tools. Phase 12 connects them into a single runAudit() call that produces actionable deviation reports — directly feeding Phase 13 (Fix Deviations).

## Output
- `src/lib/helix/audit-runner.ts` — orchestrator connecting harness → diff → validation → report
- `src/lib/helix/audit-report.ts` — report formatter producing structured markdown + JSON summaries
- Test files for both modules
</objective>

## AC-1: Audit Orchestrator Runs Full Pipeline
```gherkin
Given a set of mock scenarios and a reference corpus config
When runAudit() is called
Then it generates presets via mock harness, loads reference corpus, compares each generated preset against its family's reference schema, and returns a complete AuditResult per family
```

## AC-2: Per-Family Audit Results Include All Validation Layers
```gherkin
Given an AuditResult for a device family
When inspecting the result
Then it contains structural diff summaries (deviation counts by category/severity), intent audit results, musical audit results, and schema compliance (missing required keys) for each scenario in that family
```

## AC-3: Structured Report Generation
```gherkin
Given a complete AuditResult set
When formatAuditReport() is called
Then it produces a structured markdown report with per-family sections, deviation summaries, top issues, and an overall pass/fail status per family
```

## AC-4: Graceful Handling of Missing References
```gherkin
Given a device family with no reference presets available
When runAudit() processes that family
Then it still runs intent/musical validation and reports "no reference corpus" for structural comparison (skips diff, does not error)
```

<task type="auto">
  <name>Task 1: Build audit orchestrator (audit-runner.ts)</name>
  <files>src/lib/helix/audit-runner.ts, src/lib/helix/audit-runner.test.ts</files>
  <action>
    Create audit-runner.ts with:

    1. **Types:**
       - `AuditConfig` — { scenarios: MockScenario[], corpusConfig?: CorpusConfig }
       - `ScenarioAuditResult` — { scenarioId, device, toneStyle, harnessResult: HarnessResult, diffReport?: DiffReport, schemaCompliance?: SchemaComplianceResult }
       - `SchemaComplianceResult` — { missingRequiredKeys: string[], missingCommonKeys: string[], extraKeys: string[] }
       - `FamilyAuditResult` — { family: DeviceFamily, scenarios: ScenarioAuditResult[], diffSummary: { critical, warning, info }, intentSummary: { passed, failed }, musicalSummary: { passed, failed }, schemaCompliance: { totalMissing, details }, overallPassed: boolean }
       - `AuditResult` — { families: FamilyAuditResult[], totalScenarios, totalPassed, totalFailed, timestamp: string }

    2. **Functions:**
       - `runAudit(config: AuditConfig): AuditResult` — Main orchestrator:
         a. Run all scenarios via runAllScenarios() → HarnessResult[]
         b. If corpusConfig provided, load reference corpus via loadCorpus()
         c. Group results by device family
         d. For each family:
            - Extract family schema from reference presets (if available)
            - For each scenario result:
              - If reference presets exist for family: diffPresets(reference[0], generated, family) for structural comparison
              - Check schema compliance: compare generated preset keys against FamilySchema.requiredTopLevelKeys and blockStructure
            - Aggregate counts: critical/warning/info deviations, intent pass/fail, musical pass/fail
            - overallPassed = no critical deviations AND all intent passed AND all musical passed
         e. Return AuditResult with all families + totals

       - `checkSchemaCompliance(preset: unknown, schema: FamilySchema): SchemaComplianceResult` — Compare generated preset top-level keys and block structure against family schema's required/common keys

       - Helper: `groupByFamily(results: HarnessResult[]): Map<DeviceFamily, HarnessResult[]>` — Group using device-to-family mapping

    3. **Test file:** Test with mock data (don't call real pipeline in unit tests):
       - Test groupByFamily correctly groups all 25 scenarios into 4 families
       - Test checkSchemaCompliance detects missing required keys
       - Test checkSchemaCompliance passes when all required keys present
       - Test runAudit with no corpus config still produces results (intent/musical only)
       - Test runAudit overallPassed logic (critical deviation → fail, musical fail → fail)
       - Test graceful handling when corpus has no presets for a family

    Avoid: Importing node:fs directly in audit-runner (corpus loading handles that). Don't make runAudit async unless corpus loader requires it — check if loadCorpus is sync or async.
  </action>
