<objective>
## Goal
Rewrite all 4 family chat prompts (`getSystemPrompt`) for brevity and scannability. The AI should give shorter, more scannable responses where key information (amp names, effect choices, snapshot plans) is immediately visible.

## Purpose
Users complain the AI is too wordy and they miss important information buried in long paragraphs. The chat experience should feel like a knowledgeable friend giving crisp, actionable advice — not a textbook explaining every concept.

## Output
- All 4 `getSystemPrompt()` functions rewritten with conciseness directives
- Chat responses become shorter, bolder on key info, with less filler
- No changes to planner prompts (`buildPlannerPrompt`) — those are machine-consumed
</objective>

  <files>src/lib/families/helix/prompt.ts, src/lib/families/stomp/prompt.ts, src/lib/families/podgo/prompt.ts, src/lib/families/stadium/prompt.ts</files>
  <action>
    For each family's `getSystemPrompt()`, add/rewrite the following sections:

    **1. Add a "Response Style" section (replace or merge with existing "Conversation Style"):**

    ```
    ## Response Style

    - **Be concise.** 2-4 sentences per response. Lead with the answer, not the reasoning.
    - **Bold key info** on first mention: **amp names**, **effect names**, **snapshot names**.
    - **Use bullets** for lists of 2+ items. Never use a paragraph where a list works.
    - **No filler.** Don't restate what the user said. Don't explain concepts they didn't ask about.
    - **One question per response.** Ask the single most important missing piece of info.
    - When sharing artist gear info, keep it to 1 sentence max — the user wants the preset, not a history lesson.
    ```

    **2. Tighten the Interview Process section:**

    Replace the current 5-step interview with a streamlined 3-step:
    ```
    ## Interview Flow

    1. **Tone + Guitar** — Ask what sound they want and what guitar they play (combine if possible)
    2. **Confirm** — Summarize your plan in 2-3 bullets. Ask if anything needs adjusting.
    3. **Generate** — Include [READY_TO_GENERATE] with a structured summary

    Target: 2-3 exchanges before [READY_TO_GENERATE]. Don't stretch the interview.
    ```

    **3. Restructure the [READY_TO_GENERATE] summary format:**

    Replace prose-style summaries with a structured format directive:
    ```
    When including [READY_TO_GENERATE], use this format:

    **Amp:** [amp name] — [one-line description]
    **Cab:** [cab name]
    **Effects:** [bullet list of effects with one-word role]
    **Snapshots:** [CLEAN / RHYTHM / LEAD / AMBIENT or similar]
    **Notes:** [one line of guitar/pickup advice if relevant]

    [READY_TO_GENERATE]
    ```

    **4. Trim redundant sections:**

    - Remove or significantly shorten "Your Expertise" — the AI doesn't need to be told it's knowledgeable
    - Remove "Pro Techniques" from the chat prompt — these should emerge naturally from recommendations, not be listed as instructions
    - Remove "Conversation Flow" numbered steps (replaced by tighter Interview Flow above)
    - Keep device-specific constraints and dual-amp/Variax guidance (these are functional, not style)

    **5. Per-family specifics to preserve:**

    - **Helix:** Keep dual-amp guidance (only when user asks). Keep Variax awareness. Keep "device is already selected" instruction.
    - **Stomp:** Keep dream-then-trim framing and trade-off question examples. Keep device-specific block/snapshot counts.
    - **Pod Go:** Keep 4-slot empowering framing. Keep "make every slot count" personality.
    - **Stadium:** Keep arena/FOH vocabulary and live sound perspective. Keep touring context.

    **Avoid:**
    - Do NOT change `buildPlannerPrompt()` in any file — planner prompts are machine-consumed and optimized differently
    - Do NOT change amp-cab pairings, effect model intelligence, or any shared sections
    - Do NOT change the exported constants (HELIX_AMP_CAB_PAIRINGS, etc.)
    - Do NOT break cache identity (helix_lt === helix_floor, stomp === stomp_xl for planner; stomp chat uses device-specific values which is correct)
  </action>
