<objective>
## Goal
Build a reference corpus loader and schema extractor that reads real Line 6 preset files (.hlx/.pgp/.hsp) from disk, parses them into structured JSON, groups by device family, and extracts per-family "gold standard" schemas defining what structurally correct presets look like.

## Purpose
The diff engine (Phase 9) can compare two presets — but we need the "reference side" of that comparison. This phase builds the infrastructure to ingest real-world reference presets and extract consistent structural patterns (required keys, block configurations, controller assignments, metadata fields, snapshot structure) so Phase 12 can run full audit comparisons.

## Output
- `src/lib/helix/reference-corpus.ts` — Loader that reads preset files, parses per format, groups by DeviceFamily
- `src/lib/helix/schema-extractor.ts` — Analyzes multiple presets per family, extracts common structural patterns into a FamilySchema
- Tests for both modules
</objective>

<acceptance_criteria>

## AC-1: Reference Corpus Loader
```gherkin
Given a list of file paths grouped by device family
When loadCorpus() is called
Then each file is read, parsed (JSON for .hlx/.pgp, skip 8-byte header for .hsp), and returned as a typed ReferencePreset with family, filePath, deviceId, and parsed JSON content
And invalid/missing files are reported as errors without aborting the entire load
```

## AC-2: Device Family Detection
```gherkin
Given a parsed preset file
When the loader processes it
Then the device family is detected from the file's device ID field (data.device for .hlx, json.preset.head.device for .pgp, meta.device_id for .hsp)
And the family matches the expected grouping (helix/stomp/podgo/stadium)
```

## AC-3: Schema Extraction
```gherkin
Given 2+ parsed reference presets from the same device family
When extractFamilySchema() is called
Then it produces a FamilySchema containing:
  - requiredTopLevelKeys (keys present in ALL presets)
  - blockStructure (common block key patterns, required fields per block)
  - snapshotStructure (required snapshot fields, controller patterns)
  - controllerStructure (required controller patterns)
  - footswitchStructure (required footswitch patterns)
  - metadataFields (common metadata keys)
```

## AC-4: Multi-Preset Consensus
```gherkin
Given reference presets that vary in content (different amps, effects, etc.)
When extractFamilySchema() processes them
Then only keys/patterns present in ALL presets are marked "required"
And keys present in SOME presets are marked "common" with a frequency count
And per-preset-specific keys (like specific amp parameter names) are excluded from required patterns
```

  <name>Task 1: Build reference corpus loader</name>
  <files>src/lib/helix/reference-corpus.ts, src/lib/helix/reference-corpus.test.ts</files>
  <action>
    Create reference-corpus.ts with:

    **Types:**
    - `ReferencePreset`: { family: DeviceFamily; filePath: string; fileName: string; deviceId: number; content: unknown; format: "hlx" | "pgp" | "hsp" }
    - `CorpusConfig`: Record<DeviceFamily, string[]> — maps family to file paths
    - `CorpusLoadResult`: { presets: ReferencePreset[]; errors: { filePath: string; error: string }[] }

    **Functions:**
    - `parsePresetFile(filePath: string): { content: unknown; format: "hlx" | "pgp" | "hsp" }` — reads file, detects format from extension, parses JSON (strips 8-byte magic header for .hsp using STADIUM_CONFIG.STADIUM_MAGIC_HEADER)
    - `detectDeviceId(content: unknown, format: string): number` — extracts device ID from correct path per format:
      - hlx: content.data.device
      - pgp: content.data.tone.controller.dsp0 exists check (Pod Go uses same top-level schema as hlx but with device ID at content.data.device)
      - hsp: content.meta.device_id
    - `detectFamily(deviceId: number): DeviceFamily` — maps device IDs to families using DEVICE_IDS from types.ts
    - `loadCorpus(config: CorpusConfig): Promise<CorpusLoadResult>` — loads all configured files, returns presets + errors

    **Tests (reference-corpus.test.ts):**
    - Test parsePresetFile with real .hlx file (use one of the known reference paths)
    - Test parsePresetFile with real .pgp file
    - Test parsePresetFile with real .hsp file (verify magic header stripping)
    - Test detectDeviceId for each format
    - Test detectFamily mapping
    - Test loadCorpus with mixed families, verify grouping
    - Test error handling for missing/invalid files

    **Important:** Use fs.readFileSync (Node.js) — these are test/audit tools, not production browser code. Import from "node:fs".
    Avoid: async where not needed (file reads are fine sync for test tooling).
  </action>
  <verify>npx vitest run src/lib/helix/reference-corpus.test.ts</verify>
  <done>AC-1 and AC-2 satisfied: corpus loader reads all 3 formats, detects device family, handles errors gracefully</done>

<task type="auto">
  <name>Task 2: Build schema extractor</name>
  <files>src/lib/helix/schema-extractor.ts, src/lib/helix/schema-extractor.test.ts</files>
  <action>
    Create schema-extractor.ts with:

    **Types:**
    - `KeyFrequency`: { key: string; count: number; total: number; status: "required" | "common" | "rare" }
      - required: present in ALL presets (count === total)
      - common: present in >50% (count > total/2)
      - rare: present in <=50%
    - `BlockPattern`: { blockKeyPattern: string; requiredFields: string[]; commonFields: KeyFrequency[] }
    - `FamilySchema`: {
        family: DeviceFamily;
        presetCount: number;
        requiredTopLevelKeys: string[];
        blockStructure: BlockPattern[];
        snapshotStructure: { requiredKeys: string[]; commonKeys: KeyFrequency[] };
        controllerStructure: { requiredKeys: string[]; commonKeys: KeyFrequency[] };
        footswitchStructure: { requiredKeys: string[]; commonKeys: KeyFrequency[] };
        metadataFields: KeyFrequency[];
      }

    **Functions:**
    - `collectKeys(obj: unknown, prefix?: string): string[]` — recursively enumerate all JSON paths in a preset (e.g., "data.tone.dsp0.block0.@model")
    - `computeKeyFrequencies(allKeysets: string[][], total: number): KeyFrequency[]` — count how often each key appears across presets, classify as required/common/rare
    - `extractBlockPatterns(keyFreqs: KeyFrequency[], family: DeviceFamily): BlockPattern[]` — group keys by block pattern (e.g., "dsp0.block*"), extract per-block required/common fields
    - `extractFamilySchema(presets: ReferencePreset[]): FamilySchema` — main entry: takes presets of ONE family, returns schema
      - Collect all keys from each preset
      - Compute frequencies
      - Partition into block/snapshot/controller/footswitch/metadata using same path patterns as structural-diff.ts (METADATA_PATTERNS, BLOCK_PATTERNS, SNAPSHOT_PATTERNS, etc.)
      - Build FamilySchema

    **Tests (schema-extractor.test.ts):**
    - Test collectKeys with a small nested object
    - Test computeKeyFrequencies with overlapping keysets
    - Test extractFamilySchema with 2 synthetic presets sharing most structure
    - Test that per-preset-specific keys (like amp-specific params) are NOT marked required
    - Test with real reference presets if available (at least 2 per family)

    **Important:** Reuse path classification patterns from structural-diff.ts (import or duplicate the regex arrays). This ensures schema categories align with diff categories.
    Avoid: Making this async — pure data processing, no I/O.
  </action>
  <verify>npx vitest run src/lib/helix/schema-extractor.test.ts</verify>
  <done>AC-3 and AC-4 satisfied: schema extractor produces FamilySchema with required/common/rare classification, consensus across multiple presets</done>
