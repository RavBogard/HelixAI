<objective>
## Goal
Add rule-based intent fidelity (instrument type) and musical intelligence validation to the preset generation pipeline. All checks are deterministic — zero AI cost.

## Purpose
Phase 10 ensures generated presets (1) match what the user actually requested (intent fidelity) and (2) make musical sense (intelligence validation). This catches issues like chorus on metal, missing compression on bass, or wrong gain staging for a genre — before expensive structural comparison against references.

## Output
- Enhanced `intent-validate.ts` with instrument type checking
- New `musical-validate.ts` with genre/instrument-aware rule engine
- `MusicalAudit` type integrated into `HarnessResult`
- Tests covering all rules across 25 mock scenarios
</objective>

<task type="auto">
  <name>Task 2: Create musical-validate.ts with genre/instrument rules</name>
  <files>src/lib/helix/musical-validate.ts, src/lib/helix/musical-validate.test.ts</files>
  <action>
    Create `musical-validate.ts` following the same pattern as `quality-validate.ts`:
    - Non-throwing guarantee, always returns `MusicalWarning[]`
    - Export `MusicalWarning` interface (code, severity, message, rule)
    - Export `MusicalAudit` interface: { warnings: MusicalWarning[], passed: boolean }
    - Export `validateMusicalIntelligence(intent: ToneIntent, spec: PresetSpec): MusicalAudit`

    Rules to implement (each a private function):

    1. **GENRE_EFFECT_MISMATCH** — Genre-inappropriate effects:
       - Metal/high-gain + chorus or flanger → warn "unusual for metal"
       - Blues + heavy delay (Feedback > 0.60) → info "may overwhelm blues dynamics"
       - Ambient + no delay and no reverb → warn "ambient preset needs time effects"

    2. **BASS_COMPRESSION** — Bass without dynamics block → warn

    3. **GAIN_STAGING_MISMATCH** — Amp Drive vs tone style:
       - Clean style + first amp Drive > 0.50 → warn "too much drive for clean tone"
       - High-gain style + first amp Drive < 0.30 → info "low drive for high-gain"

    4. **SNAPSHOT_ROLE_COVERAGE** — Snapshot roles vs tone style:
       - High-gain preset without a "lead" snapshot → warn "metal/high-gain benefits from lead snapshot"
       - Ambient preset without an "ambient" snapshot → warn "ambient preset should have ambient snapshot"

    5. **EFFECT_COUNT_SANITY** — Zero user-requested effects → info "no effects requested"

    The genreHint from ToneIntent drives genre rules. Map genreHint to categories:
    - "metal", "djent", "prog metal" → high-gain category
    - "blues", "classic rock" → blues category
    - "ambient", "post-rock", "shoegaze" → ambient category
    - "pop", "country", "funk", "jazz" → clean category
    - Unknown/undefined → skip genre rules

    Tone style from mock scenarios maps directly (clean, high-gain, blues, ambient, bass).

    Tests: Cover each rule with at least positive and negative cases. Use mock ToneIntent + PresetSpec fixtures (minimal — just enough fields to trigger/not trigger each rule).
  </action>
  <verify>npm test -- src/lib/helix/musical-validate.test.ts passes</verify>
  <done>AC-2, AC-3, AC-4 satisfied: musical intelligence rules with tests</done>
</task>
