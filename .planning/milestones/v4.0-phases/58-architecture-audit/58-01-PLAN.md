- Self-contained builder files (each builder owns its format independently)
- Strict model catalog separation (no cross-device fallback, throws on miss)

- Config constants grouped by device (FIRMWARE_CONFIG, POD_GO_FIRMWARE_CONFIG, STADIUM_CONFIG, STOMP_CONFIG)

**3. Fragility Points** — Document at least 5 fragility points with exact file and line references. The research identifies these:
- Fragility 1: Guard-based branching (isPodGo/isStadium/isStomp) in chain-rules.ts, param-engine.ts, validate.ts has no compiler-enforced exhaustiveness — adding a 7th device requires manual search of ~17 guard sites
- Fragility 2: Stadium/Stomp I/O model IDs hardcoded as string literals in validate.ts AND builder files — no shared constants, invisible coupling
- Fragility 3: param-engine.ts MODEL_LOOKUPS does not include STADIUM_AMPS — findModel() silently returns undefined for Stadium amps (currently harmless but invisible)
- Fragility 4: Stadium builder sequential flowPos vs. slot-grid allocation (document any fix status after Phase 53)
- Fragility 5: Hardcoded distortion model IDs in param-engine.ts instead of constant references from models.ts

For each fragility, verify the current state by reading the actual source files. If Phase 52-53 fixed something, note that explicitly.

**4. Concrete Improvements** — A table of improvements ordered by impact vs. effort:

**5. Stadium Rebuild Discoveries** — Document what Phases 52-53 actually revealed about the shared Knowledge Layer.
