# v7.5 Recommendation Report

**Generated:** 2026-03-07
**Based on:** v7.0 Milestone Audit, integration gap analysis, deferred requirements inventory

---

## Executive Summary

v7.0 delivered the complete Interactive Signal Chain Visualizer — 7 phases, 14 plans, 38 requirements, 311+ tests. All components work and are tested. Two integration gaps prevent the visualizer from being reachable from the main chat flow. v7.5 should close these gaps first, then layer on editing quality-of-life and AI-powered suggestions.

**Recommended scope:** 8 phases across 3 tiers.

---

## Tier 1: Integration Closure (Must-Have)

These fix the v7.0 audit gaps. Without them, the visualizer is an unreachable island.

### Phase 84: Chat-to-Visualizer Integration
**Goal:** After preset generation, users can click "Edit in Visualizer" to open the interactive editor with their preset loaded.

**What it does:**
- Add "Edit in Visualizer" button to post-generation card in `page.tsx`
- Call `/api/preview` with the generated `PresetSpec`
- Hydrate Zustand store via `store.hydrate(previewResult)`
- Navigate to `/visualizer` via Next.js router

**Closes:** INT-01 (chat-to-visualizer navigation)
**Risk:** Low — all components exist; routing + button only
**Estimate:** ~300 LOC

### Phase 85: Controller/Footswitch Data Flow
**Goal:** EXP pedal badges and footswitch badges display correctly in the visualizer.

**What it does:**
- Build device file in `/api/preview` route to extract controller sections
- Call `extractControllerAssignments()` and `extractFootswitchAssignments()`
- Pass results to `hydrateVisualizerState()`
- Wire `getFootswitchForBlock()` in `SignalChainCanvas` rendering loop
- Pass `footswitchAssignment` prop to `BlockTile`

**Closes:** INT-02 (orphaned extraction functions), INT-03 (unwired FS badge prop)
**Depends on:** Phase 84
**Risk:** Low — functions exist and are tested; wiring only
**Estimate:** ~400 LOC

## Tier 2: Advanced Editing (High-Value UX)

Natural progressions that make the visualizer feel complete. All were explicitly identified as deferred requirements in v7.0.

### Phase 86: Snapshot Copy/Paste
**Goal:** Users can copy parameter overrides from one snapshot to another.

**What it does:**
- "Copy Snapshot" button in SnapshotSelectorBar
- "Paste to Snapshot N" target selector
- Store action: `copySnapshotOverrides(fromIndex, toIndex)`
- Supports copying individual parameters or entire snapshot overlay

**Requirement:** EDIT-F01
**Risk:** Low
**Estimate:** ~200 LOC

### Phase 87: Block Revert-to-Defaults
**Goal:** Users can reset a single block to its AI-generated parameters without affecting other blocks.

**What it does:**
- "Reset Block" button in ParameterEditorPane
- Reads `originalBaseBlocks` for the matching block's original parameters
- Clears all snapshot overrides for that block
- Confirmation dialog to prevent accidental resets

**Requirement:** EDIT-F02
**Risk:** Low
**Estimate:** ~150 LOC

### Phase 88: Undo/Redo Stack
**Goal:** Users can undo and redo all visualizer editing actions.

**What it does:**
- Zustand middleware that records state snapshots on mutation
- Tracks: reorderBlock, moveBlock, removeBlock, addBlock, setParameterValue, swapBlockModel, toggleBlockBypass
- Ctrl+Z / Ctrl+Shift+Z keyboard shortcuts
- Undo/Redo buttons in visualizer toolbar
- Stack depth limit (e.g., 50 actions)

**Requirement:** EDIT-F03
**Risk:** Medium — state management complexity
**Estimate:** ~400 LOC

## Tier 3: AI-Powered Suggestions (Strategic)

These add intelligence to the editing experience. Higher risk due to token costs and latency.

### Phase 89: Adaptive Cab Suggestions
**Goal:** When a user swaps an amp model, the AI suggests a matching cab.

**What it does:**
- Detect amp model swap in `swapBlockModel` action
- Call lightweight `/api/suggest-cab` endpoint with genre context + new amp
- Show suggestion toast with "Apply" / "Dismiss"
- Use existing `cabAffinity` data as fast fallback before AI call

**Requirement:** REPROMPT-F01
**Risk:** Medium — token cost per suggestion; needs rate limiting
**Estimate:** ~300 LOC + new API endpoint

### Phase 90: Effect Chain Suggestions
**Goal:** When a user removes an effect, the AI suggests alternatives that fill the sonic gap.

**What it does:**
- Detect `removeBlock` action
- Analyze remaining chain for missing sonic coverage
- Call `/api/suggest-effects` with genre + current chain
- Show multi-option suggestion panel
- User can apply a suggestion or dismiss

**Requirement:** REPROMPT-F02
**Risk:** Medium — requires careful prompt engineering
**Estimate:** ~350 LOC + new API endpoint

### Phase 91: Interactive Controller Reassignment
**Goal:** Users can reassign expression pedals and footswitches to different parameters.

**What it does:**
- Dropdown/drag UI for EXP1/EXP2 assignment to any parameter
- Footswitch assignment editor per block
- Validation: device-specific controller limits (Helix 3 EXP, Stomp 2 EXP, Pod Go 1 EXP, Stadium 0)
- Parameter type compatibility checks
- Custom min/max bounds for EXP-assigned parameters

**Requirements:** CTRL-F01, CTRL-F02
**Depends on:** Phase 85 (controller data must flow first)
**Risk:** Medium — validation complexity
**Estimate:** ~600 LOC

## What's NOT in v7.5 (Remains Out of Scope)

- Cost-aware Haiku/Sonnet routing (infrastructure concern)
- Dual-amp split/join path visualization (complex routing topology)
- Full effect combination rules in visualizer (v6.0 rules are server-side by design)

## Decision Points for You

1. **Full v7.5 (all 8 phases)?** Or split into v7.5 (integration) and v8.0 (AI features)?
2. **Undo/redo priority?** Phase 88 is highest complexity — could defer to v8.0.
3. **AI suggestions (Phases 89-90)?** Token cost per suggestion needs benchmarking. Worth a spike first?
4. **Controller editing (Phase 91)?** Large scope — could be its own milestone.
