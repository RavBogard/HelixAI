   The function must determine whether a block is a "boost" (modelId is HD2_DistMinotaur or HD2_DistScream808), "AI-added distortion" (other distortion/dynamics blocks from ToneIntent), or a standard block type. 5. **Block key generation (SNAP-05):**
   Generate block keys matching the per-DSP position numbering. For each non-cab block:
   - `"block{position}"` where position is the block's `position` field
   - Group by DSP: the blockStates keys are flat ("block0", "block1", ...) and the `buildSnapshot()` in preset-builder.ts handles the DSP separation using `buildBlockKeyMap()`.

   **Important:** The SnapshotSpec.blockStates is a flat Record<string, boolean> where keys are formatted as `dsp{N}_block{position}`. Check how preset-builder.ts consumes this. Looking at the types.ts definition, `blockStates: Record<string, boolean>` is a flat key-value map. The key format must match what `buildSnapshot()` expects.

   Based on the RESEARCH.md: "The snapshot engine should use the same sequential numbering per DSP." Use keys like `"dsp0_block0"`, `"dsp0_block1"`, `"dsp1_block0"`, etc. OR check the actual preset-builder.ts consumption pattern and match it. If preset-builder.ts uses `blockStates.dsp0?.block0`, then the SnapshotSpec should nest. But types.ts shows `blockStates: Record<string, boolean>` which is flat.

   **Resolution:** Use flat keys with a pattern the executor can verify by reading preset-builder.ts. The safest approach: use `"block{globalIndex}"` where globalIndex is 0-based across the entire chain (excluding cabs), since the preset-builder's `buildBlockKeyMap()` handles remapping to per-DSP keys. Read preset-builder.ts to confirm the exact key format expected.
