<task type="auto">
  <name>Task 1: Create AuthButton component with sign-in/sign-out and hydration safety</name>
  <files>src/components/auth/AuthButton.tsx</files>
  <action>
Create `src/components/auth/AuthButton.tsx` as a Client Component (`'use client'`):

1. **Props interface:**
   ```typescript
   interface AuthButtonProps {
     onBeforeSignIn?: () => void  // Called before OAuth redirect to serialize chat state
   }
   ```

2. **State:** `user` (null or user object), `mounted` (boolean, starts false).

3. **useEffect on mount:**
   a. Set `mounted = true`.
   b. Create Supabase browser client.
   c. Call `supabase.auth.getUser()` to set initial user state.
   d. Subscribe to `onAuthStateChange` to keep user state synced. On any auth event, update user state and call `router.refresh()` to re-render server components.
   e. Return cleanup function that unsubscribes.

4. **Sign-in handler (`handleSignIn`):**
   a. Call `props.onBeforeSignIn?.()` FIRST — this serializes chat state to sessionStorage before the redirect.
   b. Create Supabase browser client.
   c. Get current user via `supabase.auth.getUser()`.
   d. If `user?.is_anonymous` is true:
      - Call `supabase.auth.linkIdentity({ provider: 'google', options: { redirectTo: \`${window.location.origin}/auth/callback\` } })`.
      - This preserves the anonymous UUID. Do NOT use `signInWithOAuth()`.
   e. If no user exists at all (edge case — session was not created by page.tsx):
      - Fall back to `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: \`${window.location.origin}/auth/callback\` } })`.
   f. Log errors to console but do not throw — the redirect will handle error display via auth_error param.

5. **Sign-out handler (`handleSignOut`):**
   a. Create Supabase browser client.
   b. Call `await supabase.auth.signOut()`.
   c. Immediately call `await supabase.auth.signInAnonymously()` — this restores an anonymous session so the app always has a user ID.
   d. Call `router.refresh()` to force server components to re-render.

6. **Render logic:**
   a. If `!mounted`: return a neutral placeholder `<div className="w-8 h-8" aria-hidden />` — same dimensions as the final button to prevent layout shift. This prevents hydration mismatch.
   b. If user is anonymous (`user?.is_anonymous ?? true`):
      - Render a "Sign in with Google" button styled to match the app's design system. Use the app's CSS variables (--hlx-elevated, --hlx-border, --hlx-text, etc.).
      - Button text: "Sign in" (short, fits the header). No Google logo icon needed for v2.0.
      - onClick: `handleSignIn`.
   c. If user is authenticated (`!user?.is_anonymous`):
      - Show the user's avatar from `user.user_metadata?.avatar_url`. If null, show a fallback initial from `user.email` (first character, uppercased) in a colored circle.
      - Show a "Sign out" button or make the avatar a dropdown trigger.
      - For v2.0 simplicity: show avatar + "Sign out" text button side by side. No dropdown menu.
      - onClick sign-out: `handleSignOut`.

7. **Styling:** Use Tailwind classes consistent with the existing app. The button should sit in the top-right header area. Use text-[var(--hlx-text)], bg-[var(--hlx-elevated)], border-[var(--hlx-border)], rounded-lg, text-sm, px-3, py-1.5 for the sign-in button. The avatar should be 28x28 rounded-full.

Import `useRouter` from `next/navigation`, `useState` and `useEffect` from `react`, `createSupabaseBrowserClient` from `@/lib/supabase/client`.

<task type="auto">
  <name>Task 2: Integrate AuthButton into layout.tsx and wire state serialization in page.tsx</name>
  <files>src/app/layout.tsx, src/app/page.tsx</files>
  <action>
1. **Modify `src/app/layout.tsx`:**
   - The layout is a Server Component. It cannot directly render a Client Component that depends on auth state without careful consideration. Since AuthButton manages its own state via useEffect (no server-side data dependency), it can be imported and rendered directly.
   - Import `AuthButton` from `@/components/auth/AuthButton`.
   - Add a minimal header bar above `{children}` inside the `<body>`:
     ```tsx
     <header className="fixed top-0 right-0 z-50 p-3">
       <AuthButton />
     </header>
     ```
   - Position: fixed top-right, z-50 so it floats above the main content. Minimal padding. Transparent background — the grain/ambient overlays are already behind the content.
   - Note: AuthButton receives NO props from layout.tsx — it cannot receive serializeChatState from here because layout is a Server Component and doesn't have access to page.tsx state.

2. **Modify `src/app/page.tsx` to pass serializeChatState to AuthButton:**
   - The problem: layout.tsx renders AuthButton, but serializeChatState lives in page.tsx state. Layout cannot pass page state to its own children components.
   - Solution: Use a global event pattern. Instead of passing serializeChatState as a prop, have page.tsx register a handler on `window` that AuthButton dispatches before sign-in.

   In page.tsx, add a useEffect that registers a custom event listener:
   ```typescript
   useEffect(() => {
     const handler = () => serializeChatState()
     window.addEventListener('helixai:before-signin', handler)
     return () => window.removeEventListener('helixai:before-signin', handler)
   }, [serializeChatState])
   ```

   In AuthButton.tsx, update handleSignIn to dispatch the event instead of calling a prop:
   ```typescript
   // Before OAuth redirect, dispatch event so page.tsx serializes state
   window.dispatchEvent(new Event('helixai:before-signin'))
   ```

   This decouples AuthButton (in layout) from page.tsx state without requiring a React Context provider (which would be overengineered for a single function call).

   Remove the `onBeforeSignIn` prop from AuthButton since it now uses the event pattern.

3. **Do NOT modify** any other part of layout.tsx (fonts, metadata, grain/ambient overlays, the {children} rendering).
