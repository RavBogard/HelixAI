3. Create `src/lib/supabase/server.ts` — the server client factory:
   - Import `createServerClient` from `@supabase/ssr`
   - Import `cookies` from `next/headers`
   - Export `async function createSupabaseServerClient()` that:
     a. Calls `const cookieStore = await cookies()`
     b. Returns `createServerClient(URL!, ANON_KEY!, { cookies: { getAll() { return cookieStore.getAll() }, setAll(cookiesToSet) { try { cookiesToSet.forEach(({ name, value, options }) => cookieStore.set(name, value, options)) } catch { /* Server Components cannot write cookies; middleware handles this */ } } } })`
   - The try/catch on setAll is REQUIRED because Server Components cannot write cookies. Route Handlers CAN write cookies, so the same utility works for both — the try/catch only catches in the read-only Server Component case.
   - This function is `async` because `cookies()` is async in Next.js 16.

Both files reference `process.env.NEXT_PUBLIC_SUPABASE_URL!` and `process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!`. These env vars will be configured by the user in Plan 03.

<task type="auto">
  <name>Task 2: Create middleware updateSession helper and root middleware.ts</name>
  <files>src/lib/supabase/middleware.ts, middleware.ts</files>
  <action>
1. Create `src/lib/supabase/middleware.ts` — the updateSession helper:
   - Import `createServerClient` from `@supabase/ssr`
   - Import `NextResponse, type NextRequest` from `next/server`
   - Export `async function updateSession(request: NextRequest)` that:
     a. Creates `let supabaseResponse = NextResponse.next({ request })`
     b. Creates a Supabase server client with cookie handlers that:
        - `getAll()` reads from `request.cookies.getAll()`
        - `setAll(cookiesToSet)` does the DOUBLE-WRITE pattern:
          1. Sets each cookie on `request.cookies` (for downstream server components in the same request)
          2. Recreates `supabaseResponse = NextResponse.next({ request })` (to propagate updated request cookies)
          3. Sets each cookie on `supabaseResponse.cookies` (for the browser response)
     c. Calls `await supabase.auth.getUser()` — MUST use getUser() not getSession(). getUser() contacts the Supabase auth server to verify and refresh the JWT. getSession() only reads the unverified cookie.
     d. Returns `supabaseResponse`
   - The double-write pattern on setAll is CRITICAL. Without it, refreshed tokens are not propagated to both the server-side rendering context and the browser. See 24-RESEARCH.md Pitfall 4.

2. Create `middleware.ts` at the project root (NOT inside src/):
   - Import `updateSession` from `@/lib/supabase/middleware`
   - Import `type NextRequest` from `next/server`
   - Export `async function middleware(request: NextRequest)` that calls `return await updateSession(request)`
   - Export `const config = { matcher: ['/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)'] }`
   - This matcher runs middleware on ALL routes (including /api/*) except static assets. This is CORRECT — middleware only refreshes session tokens; it never blocks requests. Anonymous API requests pass through with no session to refresh, adding negligible overhead.
   - Do NOT add any redirect or blocking logic to middleware. Each API route independently decides what to do with the session (or lack thereof). This preserves the existing anonymous flow at zero performance cost.
  </action>
