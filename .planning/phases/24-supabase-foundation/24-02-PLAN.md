  <action>
Create `supabase/schema.sql` at the project root (create the `supabase/` directory). This file is run manually by the user in the Supabase SQL Editor. It is NOT an automated migration — it is a one-time schema setup script.

The SQL must contain, in this exact order:

1. **conversations table:**
   ```sql
   CREATE TABLE conversations (
     id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     user_id     UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
     title       TEXT NOT NULL DEFAULT 'New Chat',
     device      TEXT NOT NULL,
     preset_url  TEXT,
     created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
     updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
   );
   ```
   - `device` stores 'helixLT' | 'helixFloor' | 'podGo' as text (no CHECK constraint — flexibility for future devices)
   - `preset_url` is nullable — null until first preset generation in a conversation
   - `user_id` references auth.users with CASCADE delete — when a user is deleted, all their conversations are removed

2. **Enable RLS on conversations IMMEDIATELY after CREATE TABLE:**
   ```sql
   ALTER TABLE conversations ENABLE ROW LEVEL SECURITY;
   ```

3. **RLS policy for conversations:**
   ```sql
   CREATE POLICY "users_own_conversations"
     ON conversations FOR ALL
     USING (auth.uid() = user_id)
     WITH CHECK (auth.uid() = user_id);
   ```
   - `FOR ALL` covers SELECT, INSERT, UPDATE, DELETE
   - `USING` filters existing rows (SELECT/DELETE)
   - `WITH CHECK` filters new/modified rows (INSERT/UPDATE)

4. **messages table:**
   ```sql
   CREATE TABLE messages (
     id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
     role            TEXT NOT NULL CHECK (role IN ('user', 'assistant')),
     content         TEXT NOT NULL,
     sequence_number INTEGER NOT NULL,
     created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
   );
   ```
   - `sequence_number` is server-assigned (never client-generated timestamps) for correct ordering
   - `role` CHECK constraint prevents invalid role values
   - CASCADE delete removes messages when their conversation is deleted

5. **Composite index for message ordering:**
   ```sql
   CREATE INDEX idx_messages_conversation_sequence
     ON messages (conversation_id, sequence_number);
   ```

6. **Enable RLS on messages IMMEDIATELY:**
   ```sql
   ALTER TABLE messages ENABLE ROW LEVEL SECURITY;
   ```

7. **RLS policy for messages (via conversation ownership):**
   ```sql
   CREATE POLICY "users_own_messages"
     ON messages FOR ALL
     USING (
       conversation_id IN (
         SELECT id FROM conversations WHERE user_id = auth.uid()
       )
     )
     WITH CHECK (
       conversation_id IN (
         SELECT id FROM conversations WHERE user_id = auth.uid()
       )
     );
   ```
   - Messages are owned through their parent conversation — no separate user_id column on messages
   - The subquery checks that the message's conversation belongs to the current user

8. **Storage bucket creation:**
   ```sql
   INSERT INTO storage.buckets (id, name, public)
   VALUES ('presets', 'presets', false);
   ```
   - Private bucket (public: false) — files are only accessible via signed URLs

9. **Storage RLS policies (4 policies for INSERT, SELECT, UPDATE, DELETE):**
   - Path convention: `{user_id}/{conversation_id}/latest.hlx` (or `.pgp`)
   - `storage.foldername(name)` is 1-indexed: index [1] = user_id, index [2] = conversation_id
   - All policies check `(storage.foldername(name))[1] = auth.uid()::text` for user isolation
   - All policies filter on `bucket_id = 'presets'`
   - All policies use `TO authenticated` role (not anon — only authenticated users can store presets)
   - Write the 4 storage policies exactly as shown in 24-RESEARCH.md Pattern 5 (users_upload_own_presets, users_read_own_presets, users_update_own_presets, users_delete_own_presets)

1. Create `src/app/api/cron/keep-alive/route.ts`:
   - Import `NextRequest, NextResponse` from `next/server`
   - Import `createSupabaseServerClient` from `@/lib/supabase/server`
   - Export `const dynamic = 'force-dynamic'` (prevents caching of cron responses)
   - Export `async function GET(request: NextRequest)`:
     a. Check authorization header: `request.headers.get('authorization') !== 'Bearer ' + process.env.CRON_SECRET'` → return 401 Unauthorized
     b. Call `const supabase = await createSupabaseServerClient()`
     c. Execute a REAL database query — NOT an HTTP ping:
        ```typescript
        const { error } = await supabase
          .from('conversations')
          .select('id')
          .limit(1)
        ```
        This query touches the conversations table, which registers as database activity and prevents the 7-day inactivity pause. An HTTP-only ping does NOT reliably prevent pauses (see 24-RESEARCH.md Pitfall 3).
     d. If error: return `NextResponse.json({ ok: false, error: error.message }, { status: 500 })`
     e. If success: return `NextResponse.json({ ok: true, timestamp: new Date().toISOString() })`
   - The CRON_SECRET validation prevents unauthorized calls to this endpoint. Vercel automatically sends the CRON_SECRET as a Bearer token in cron requests.

2. Create `vercel.json` at the project root:
   ```json
   {
     "crons": [
       {
         "path": "/api/cron/keep-alive",
         "schedule": "0 9 */4 * *"
       }
     ]
   }
   ```
   - Schedule: `0 9 */4 * *` = every 4 days at 09:00 UTC
   - This provides a 3-day safety buffer against the 7-day pause threshold
   - Vercel Hobby plan supports cron jobs (no paid plan required)
   - If a vercel.json already exists, MERGE the crons array into it. Do not overwrite other config.
