**Part B — Fix validate.ts:**

Read validate.ts fully before editing. Find and fix two issues:

**Fix 1 — Mic range clamping:**
Find any code that clamps Mic to a range of 0-7 (such as `Math.min(7, ...)` or `Math.max(0, Math.min(7, ...))`). Change the upper bound from 7 to 15.

Reason: Real HX Edit exports contain `Mic: 11` (Neumann U67 Condenser). The current clamp of 7 would silently corrupt a valid mic selection. The exact upper bound is unconfirmed but 15 is safe based on observed values up to 11.

**Fix 2 — LowCut/HighCut range checks:**
Find any validation code that checks LowCut or HighCut against a range of 0-1 (e.g., `>= 0 && <= 1` or `< 0` or `> 1`). Update these checks to use Hz ranges:
- LowCut valid range: 19.9 to 500.0 (Hz) — anything below 1.0 is certainly a normalized-float bug
- HighCut valid range: 1000.0 to 20100.0 (Hz) — anything below 1.0 is certainly wrong

If the validator has auto-correction logic that sets LowCut or HighCut to a default value on invalid input, update the defaults to Hz values:
- LowCut default: 80.0 (not 0.18)
- HighCut default: 8000.0 (not 0.80)

If validate.ts does not currently check LowCut/HighCut at all, add a simple range check:
```typescript
// Validate cab LowCut and HighCut are in Hz (not normalized float)
if (block.parameters.LowCut !== undefined && block.parameters.LowCut < 19.9) {
  console.warn(`[validate] Cab block has LowCut=${block.parameters.LowCut} — looks like normalized float, should be Hz (e.g., 80.0). Correcting to 80.0`);
  block.parameters.LowCut = 80.0;
}
if (block.parameters.HighCut !== undefined && block.parameters.HighCut < 100.0) {
  console.warn(`[validate] Cab block has HighCut=${block.parameters.HighCut} — looks like normalized float, should be Hz (e.g., 8000.0). Correcting to 8000.0`);
  block.parameters.HighCut = 8000.0;
}
```

After editing both files, run `npx tsc --noEmit` to verify zero new TypeScript errors.
