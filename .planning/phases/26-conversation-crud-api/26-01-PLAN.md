Create `src/app/api/conversations/route.ts` (create the `src/app/api/conversations/` directory).

This file exports two handlers: POST (create conversation) and GET (list conversations).

**POST handler â€” create conversation (CONV-01, CONV-04):**
1. Import `NextRequest, NextResponse` from `next/server` and `createSupabaseServerClient` from `@/lib/supabase/server`.
2. Call `const supabase = await createSupabaseServerClient()`.
3. Call `const { data: { user }, error: authError } = await supabase.auth.getUser()`. If `authError` or `!user`, return `NextResponse.json({ error: 'Unauthorized' }, { status: 401 })`. This is the defense-in-depth session verification per STATE.md decision â€” every route handler independently verifies session, never relying solely on middleware.
4. Parse body: `const { device } = await request.json()`.
5. Validate `device` exists and is a non-empty string. If missing, return `NextResponse.json({ error: 'device is required' }, { status: 400 })`.
6. Insert into conversations table:
   ```typescript
   const { data, error } = await supabase
     .from('conversations')
     .insert({ user_id: user.id, device, title: 'New Chat' })
     .select('id, title, device, created_at, updated_at')
     .single()
   ```
   Note: `user_id: user.id` is set explicitly even though RLS would reject inserts with a different user_id. This is defense-in-depth.
7. If error, return `NextResponse.json({ error: error.message }, { status: 500 })`.
8. Return `NextResponse.json(data, { status: 201 })` â€” 201 Created, not 200.

**GET handler â€” list conversations (CONV-05):**
1. Same auth verification pattern as POST.
2. Query:
   ```typescript
   const { data, error } = await supabase
     .from('conversations')
     .select('id, title, device, updated_at, created_at')
     .eq('user_id', user.id)
     .order('updated_at', { ascending: false })
     .limit(50)
   ```
   The `.eq('user_id', user.id)` is explicit defense-in-depth on top of RLS. The limit of 50 prevents unbounded queries.
3. If error, return 500. Otherwise return `NextResponse.json(data ?? [])` â€” return empty array if data is null, not null itself.

**File 1: `src/app/api/conversations/[id]/route.ts` â€” GET read with messages (CONV-05 read detail):**

Create the `src/app/api/conversations/[id]/` directory.

1. Import `NextRequest, NextResponse` from `next/server` and `createSupabaseServerClient` from `@/lib/supabase/server`.
2. Export GET handler with Next.js App Router dynamic route params. In Next.js 15+, `params` is a Promise:
   ```typescript
   export async function GET(
     request: NextRequest,
     { params }: { params: Promise<{ id: string }> }
   ) {
     const { id } = await params
   ```
3. Auth verification: same pattern â€” `supabase.auth.getUser()`, reject if no user (401).
4. Validate `id` is a non-empty string.
5. Use Supabase embedded select with referenced table ordering for a single round trip:
   ```typescript
   const { data, error } = await supabase
     .from('conversations')
     .select(`
       id, title, device, preset_url, created_at, updated_at,
       messages(id, role, content, sequence_number, created_at)
     `)
     .eq('id', id)
     .eq('user_id', user.id)
     .order('sequence_number', { referencedTable: 'messages', ascending: true })
     .single()
   ```
   The `.eq('user_id', user.id)` verifies ownership. Combined with `.single()`, if the user does not own this conversation (or it does not exist), the query returns an error.
6. If error or `!data`, return `NextResponse.json({ error: 'Not found' }, { status: 404 })`. ALWAYS return 404 for both "not found" and "not owned" â€” per Pitfall 4 in research, never leak existence to non-owners by returning 403.
7. If success, return `NextResponse.json(data)`.

**File 2: `src/app/api/conversations/[id]/title/route.ts` â€” PATCH title update (CONV-02):**

Create the `src/app/api/conversations/[id]/title/` directory.

1. Same imports.
2. Export PATCH handler with Promise params:
   ```typescript
   export async function PATCH(
     request: NextRequest,
     { params }: { params: Promise<{ id: string }> }
   ) {
     const { id } = await params
   ```
3. Auth verification: same pattern.
4. Parse body: `const { title } = await request.json()`.
5. Validate `title` is a non-empty string: `if (!title || typeof title !== 'string')` return 400 with `{ error: 'title is required' }`.
6. Update:
   ```typescript
   const { data, error } = await supabase
     .from('conversations')
     .update({ title: title.trim().slice(0, 60), updated_at: new Date().toISOString() })
     .eq('id', id)
     .eq('user_id', user.id)
     .select('id, title, updated_at')
     .single()
   ```
   Title is trimmed and truncated to 60 characters server-side (CONV-02 says "5-8 words" â€” 60 chars is a safe maximum). The `.eq('user_id', user.id)` ensures ownership check.
7. If error or `!data`, return 404 (same non-leakage pattern).
8. If success, return `NextResponse.json(data)`.

**Important â€” Next.js 15+ params handling:** In Next.js 15+, the `params` object in route handlers is a Promise. You MUST `await params` before accessing `.id`. If you destructure synchronously (`{ params: { id } }`), it will fail at runtime with a "params is a promise" error.
