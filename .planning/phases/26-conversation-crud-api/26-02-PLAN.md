This file exports one handler: POST (save a message to a conversation).

**POST handler — save message (CONV-03):**

1. Import `NextRequest, NextResponse` from `next/server` and `createSupabaseServerClient` from `@/lib/supabase/server`.

2. Export POST handler with Next.js 15+ Promise params:
   ```typescript
   export async function POST(
     request: NextRequest,
     { params }: { params: Promise<{ id: string }> }
   ) {
     const { id: conversationId } = await params
   ```

3. Auth verification: call `supabase.auth.getUser()`, reject if no user (401).

4. Verify conversation ownership BEFORE saving the message — this prevents saving messages to conversations owned by other users:
   ```typescript
   const { data: conversation, error: convError } = await supabase
     .from('conversations')
     .select('id')
     .eq('id', conversationId)
     .eq('user_id', user.id)
     .single()

   if (convError || !conversation) {
     return NextResponse.json({ error: 'Not found' }, { status: 404 })
   }
   ```

5. Parse body: `const { role, content } = await request.json()`.

6. Validate inputs:
   - `role` must be `'user'` or `'assistant'` — if not, return 400 with `{ error: 'role must be user or assistant' }`
   - `content` must be a non-empty string — if not, return 400 with `{ error: 'content is required' }`

7. Compute server-side sequence number. This is the critical CONV-03 implementation — sequence numbers are NEVER client-generated (STATE.md locked decision):
   ```typescript
   const { data: maxSeq } = await supabase
     .from('messages')
     .select('sequence_number')
     .eq('conversation_id', conversationId)
     .order('sequence_number', { ascending: false })
     .limit(1)
     .single()

   const nextSeq = (maxSeq?.sequence_number ?? 0) + 1
   ```
   This reads the current MAX sequence_number for the conversation. If no messages exist yet, `maxSeq` is null and `nextSeq` defaults to 1. The composite index on `(conversation_id, sequence_number)` makes this query fast.

8. Insert the message:
   ```typescript
   const { data: message, error: insertError } = await supabase
     .from('messages')
     .insert({
       conversation_id: conversationId,
       role,
       content,
       sequence_number: nextSeq,
     })
     .select('id, conversation_id, role, content, sequence_number, created_at')
     .single()
   ```

9. If insert error, return 500.

10. Update `conversations.updated_at` to reflect this new activity — this keeps the list ordering correct (most recent conversation first in GET /api/conversations):
    ```typescript
    await supabase
      .from('conversations')
      .update({ updated_at: new Date().toISOString() })
      .eq('id', conversationId)
      .eq('user_id', user.id)
    ```
    This update is fire-and-forget for the response — if it fails, the message was still saved. Do not block the 201 response on this update.

11. Return `NextResponse.json(message, { status: 201 })`.

**Concurrency note (DO NOT pre-solve):** In the extremely rare case of two concurrent saves to the same conversation, both could read the same MAX and produce duplicate sequence numbers. This is acceptable for this single-user sequential chat tool per the research analysis. A DB trigger would prevent it but requires schema changes. Do not add advisory locks or retry logic.

**DELETE handler — delete conversation with storage cleanup (CONV-06):**

1. Add the DELETE export to the existing file. The GET handler remains unchanged.

2. The DELETE handler follows a specific three-step ordering that MUST be respected:

   **Step 1 — Read preset_url and verify ownership:**
   ```typescript
   export async function DELETE(
     request: NextRequest,
     { params }: { params: Promise<{ id: string }> }
   ) {
     const { id } = await params
     const supabase = await createSupabaseServerClient()
     const { data: { user }, error: authError } = await supabase.auth.getUser()
     if (authError || !user) {
       return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
     }

     // Fetch conversation to get preset_url BEFORE deleting
     const { data: conversation, error: fetchError } = await supabase
       .from('conversations')
       .select('id, preset_url')
       .eq('id', id)
       .eq('user_id', user.id)
       .single()

     if (fetchError || !conversation) {
       return NextResponse.json({ error: 'Not found' }, { status: 404 })
     }
   ```
   CRITICAL: The preset_url MUST be read BEFORE the conversation row is deleted. Once the row is deleted, the URL is gone forever. This is called out in Research Pitfall 2.

   **Step 2 — Delete preset file from Storage (non-fatal):**
   ```typescript
     if (conversation.preset_url) {
       // preset_url stores the storage object path (e.g., "{user_id}/{conv_id}/latest.hlx")
       // NOT a full HTTPS URL — this is the Phase 26/27 contract
       await supabase.storage.from('presets').remove([conversation.preset_url])
       // Storage delete errors are intentionally ignored — the DB delete must proceed
       // The file may already be gone (manual cleanup, storage error, etc.)
     }
   ```
   Storage errors are NON-FATAL. If the file does not exist, if the path is wrong, if storage is down — the conversation delete MUST still proceed. Never let a storage failure orphan a conversation row in the database.

   **Step 3 — Delete conversation row (cascades to messages):**
   ```typescript
     const { error: deleteError } = await supabase
       .from('conversations')
       .delete()
       .eq('id', id)
       .eq('user_id', user.id)

     if (deleteError) {
       return NextResponse.json({ error: deleteError.message }, { status: 500 })
     }

     return NextResponse.json({ ok: true })
   }
   ```
   The FK cascade on `messages.conversation_id` automatically deletes all messages for this conversation. No explicit message deletion is needed.

**Contract for Phase 27:** The `preset_url` column stores the Supabase Storage object path (e.g., `abc123-user-id/def456-conv-id/latest.hlx`), NOT a full HTTPS URL. Phase 27's preset upload code MUST write the path in this format. The `supabase.storage.from('presets').remove([path])` method takes the object path, not a URL.
