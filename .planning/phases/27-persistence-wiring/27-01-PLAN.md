<objective>
Wire the existing `/api/chat` and `/api/generate` route handlers to conditionally persist messages and preset files when a `conversationId` is present in the request body.

Purpose: These two route modifications are the core persistence layer — without them, conversations exist as empty database rows. This plan makes the existing stateless routes persistence-aware while guaranteeing zero behavioral change for anonymous flows (UXP-04). The server routes must be wired first so the client (Plan 02) has working persistence endpoints to call.

Output: Modified `chat/route.ts` and `generate/route.ts` that accept an optional `conversationId`, persist data to Supabase when provided, and behave identically to v1.3 when it is absent.
</objective>

**Step 4 — Save user message BEFORE streaming (gated):**
After the persistence variable declarations, before the `const ai = createGeminiClient();` line, add the following block. This saves the last user message to the messages table BEFORE opening the Gemini stream, so user messages are persisted even if the stream fails:

```typescript
if (conversationId) {
  supabase = await createSupabaseServerClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (user) {
    userId = user.id;
    const lastMsg = messages[messages.length - 1];
    if (lastMsg?.role === "user") {
      // Verify conversation ownership — defense-in-depth (STATE.md locked decision)
      const { data: conv } = await supabase
        .from("conversations")
        .select("id")
        .eq("id", conversationId)
        .eq("user_id", userId)
        .single();

      if (conv) {
        // Server-side sequence number assignment (STATE.md locked decision)
        const { data: maxSeq } = await supabase
          .from("messages")
          .select("sequence_number")
          .eq("conversation_id", conversationId)
          .order("sequence_number", { ascending: false })
          .limit(1)
          .single();

        const nextSeq = (maxSeq?.sequence_number ?? 0) + 1;

        await supabase.from("messages").insert({
          conversation_id: conversationId,
          role: "user",
          content: lastMsg.content,
          sequence_number: nextSeq,
        });

        // Update conversations.updated_at for correct list ordering
        await supabase
          .from("conversations")
          .update({ updated_at: new Date().toISOString() })
          .eq("id", conversationId)
          .eq("user_id", userId);
      }
    }
  }
}
```

**Step 5 — Accumulate full content during streaming:**
Inside the `ReadableStream`'s `start()` callback, the route already builds the SSE stream. Add a `let fullContent = ""` accumulator before the `for await` loop, and `fullContent += text` inside the `if (text)` branch:

```typescript
const readable = new ReadableStream({
  async start(controller) {
    let fullContent = "";
    try {
      for await (const chunk of stream) {
        const text = chunk.text;
        if (text) {
          fullContent += text;
          controller.enqueue(encoder.encode(`data: ${JSON.stringify({ text })}\n\n`));
        }
      }
      controller.enqueue(encoder.encode("data: [DONE]\n\n"));
      controller.close();
```

**Step 6 — Save assistant message AFTER stream closes (fire-and-forget):**
Immediately after `controller.close()` inside the try block, add the assistant message save. This is fire-and-forget because the stream has already closed — the client received `[DONE]`:

```typescript
      // Persistence: save assistant message after stream closes
      if (conversationId && userId && supabase && fullContent) {
        const { data: maxSeq } = await supabase
          .from("messages")
          .select("sequence_number")
          .eq("conversation_id", conversationId)
          .order("sequence_number", { ascending: false })
          .limit(1)
          .single();

        supabase
          .from("messages")
          .insert({
            conversation_id: conversationId,
            role: "assistant",
            content: fullContent,
            sequence_number: (maxSeq?.sequence_number ?? 0) + 1,
          })
          .then(({ error }) => {
            if (error) console.error("Failed to save assistant message:", error.message);
          });
      }
```

CRITICAL — the `.then()` pattern (NOT `await`) is intentional. The stream is already closed. The client does not wait for this DB write. Making it fire-and-forget prevents any latency addition to the chat UX.

CRITICAL — DO NOT write messages per-chunk inside the streaming loop. Research Pitfall 1 explains why: it creates 20-200 tiny INSERTs, holds a DB connection open for 3-20 seconds, and leaves partial messages on interruption.

For the Pod Go branch (before `return NextResponse.json({ preset: pgpFile, ... })`):
```typescript
    // --- Persistence: fire-and-forget upload + preset_url update (STORE-01, STORE-03) ---
    if (conversationId) {
      const supabase = await createSupabaseServerClient();
      const { data: { user } } = await supabase.auth.getUser();
      if (user) {
        const storagePath = `${user.id}/${conversationId}/latest.pgp`;
        const fileBuffer = Buffer.from(JSON.stringify(pgpFile));

        // Non-blocking — do NOT await before return
        supabase.storage
          .from("presets")
          .upload(storagePath, fileBuffer, {
            contentType: "application/json",
            upsert: true,

            if (!uploadError) {
              return supabase
                .from("conversations")
                .update({ preset_url: storagePath, updated_at: new Date().toISOString() })
                .eq("id", conversationId)
                .eq("user_id", user.id);
            }
            console.error("Preset storage upload failed (non-fatal):", uploadError.message);
          })
          .catch((err) => console.error("Preset persistence error (non-fatal):", err));
      }
    }
    // --- End persistence ---

    return NextResponse.json({ ... });
```

**CRITICAL — preset_url stores the object path, NOT a full HTTPS URL.** The Phase 26 DELETE handler calls `supabase.storage.from('presets').remove([conversation.preset_url])` which takes object paths. If a full URL is stored here, the DELETE handler will fail silently and orphan files in Storage (Research Pitfall 5).

**CRITICAL — DO NOT await the storage upload before returning the response.** The `.then().catch()` pattern ensures the upload runs in the background. The `return NextResponse.json(...)` executes immediately. This is STORE-03: "download completes immediately from the builder output; storage upload happens asynchronously and never blocks the user."

**CRITICAL — DO NOT modify the response payload.** The `NextResponse.json(...)` returns the exact same object as v1.3. No `conversationId` echo, no `storageUrl` field, no new keys. This is UXP-04.
